## 1. 产品愿景

**一句话：** 让家庭健康数据管理像记笔记一样简单，让 AI 成为你的私人健康顾问——数据始终留在你手中。

**核心价值主张：**
- 数据主权：所有健康数据默认存储在设备本地，用户掌控自己的数据
- 家庭健康：一个设备，管理全家健康档案
- AI 赋能：将医学术语变成普通人能看懂、能行动的建议

---

## 2. 目标用户

| 用户类型 | 特征 | 核心痛点 |
|---------|------|---------|
| 家庭健康管理者 | 30-50岁，负责管理家庭成员健康 | 体检报告看不懂，历史记录散乱 |
| 慢病患者/家属 | 需要长期追踪多项健康指标 | 指标变化趋势难以掌握，就诊前准备繁琐 |
| 健康意识较强的年轻人 | 关注日常健康数据，有 Apple Watch | HealthKit 数据零散，无综合分析 |

---

## 3. 关键决策说明

### 3.1 家庭成员管理是否进入 MVP？

**决策：✅ 进入 MVP（Phase 1）**

理由：
- 家庭管理是本产品的核心差异化定位，缺失则产品价值大幅削弱
- 技术实现代价低（SwiftData 多实体关系）
- 若 Phase 1 不包含，Phase 2 添加会需要大量数据迁移工作
- 接受标准：支持最少 1 个、最多 10 个家庭成员档案的创建、编辑和删除

### 3.2 Face ID 认证 vs 简单 PIN 码？

**决策：✅ Face ID / Touch ID + 备选 6 位 PIN**

理由：
- iOS 的 LocalAuthentication 框架原生支持生物认证，开发成本极低
- 健康数据敏感性高，生物认证是行业最佳实践（参考医疗 App）
- PIN 作为生物认证失败时的降级方案，保证可用性
- 不支持简单密码（太弱），不支持强制密码（摩擦太大）

### 3.3 AI 功能是否在 Phase 1 就有基础版本？

**决策：❌ Phase 1 不包含 AI 功能（已按计划在 Phase 3 完整交付）**

理由：
- Phase 1 目标是完整的本地数据管理闭环，AI 是增强层而非基础层
- 过早引入 Claude API 会增加 Phase 1 的风险和复杂度
- 用户必须先有数据，AI 才有分析价值
- AI 功能统一在 Phase 3 引入，避免功能碎片化
- **已交付：** Phase 3 完整交付体检报告解读、趋势分析、就诊准备助手、药物识别、每日健康计划五大 AI 功能，均基于 claude-sonnet-4-6 模型

---

## 4. MVP 范围确认（Phase 1 最小可用功能集）

Phase 1 的 MVP 是"能用的本地健康档案管理 App"，核心衡量标准：
> **用户能否在没有网络的情况下完整管理一个家庭成员的健康档案？**

### Phase 1 必须有（P0）— 已全部交付 ✅

1. 应用初始化 + 本地生物认证（Face ID / Touch ID / PIN）
2. 家庭成员管理（创建、编辑、删除、切换）
3. 健康档案基础信息（基本信息 Profile）
4. 健康记录录入（手动输入，支持至少 4 种记录类型：体检记录、用药记录、就医记录、日常追踪）
5. 健康记录查看列表（按时间排序，支持按记录类型筛选）
6. 数据本地持久化（SwiftData，应用重启数据不丢失）

#### US-1.1 应用认证
**As a** 用户，
**I want** 首次打开 App 时设置认证方式，之后每次打开需通过认证，
**So that** 我的家庭健康数据不会被他人访问。

**验收标准：**
- [x] 首次启动引导用户开启 Face ID / Touch ID 认证
- [x] 若设备不支持生物认证，提供 6 位 PIN 设置
- [x] 生物认证失败 3 次后自动降级为 PIN 输入
- [x] 认证成功后进入主界面，失败则留在锁屏
- [x] App 进入后台超过 5 分钟后重新进入需再次认证
- [x] 认证界面不显示任何健康数据

**优先级：** P0 | **状态：** ✅ 已交付

## 7. 技术约束说明

### 7.1 隐私与数据安全

| 约束 | 要求 |
|------|------|
| 数据存储 | 所有健康数据存储在设备本地（SwiftData），不上传任何服务器 |
| AI 数据传输 | 调用 Claude API（claude-sonnet-4-6）时，仅传送结构化指标数据，不传图片和身份信息 |
| 用户告知 | 首次使用 AI 功能时展示数据使用说明，用户主动确认后才调用 |
| API Key 存储 | Claude API Key 由用户自行提供，通过 iOS Keychain（KeychainService）加密存储 |
| iCloud 同步 | 默认关闭，用户主动开启，v1.1 规划 |
| 认证保护 | LocalAuthentication 框架 + iOS Data Protection 文件系统加密 |

### 7.2 离线能力

| 功能 | 离线可用 |
|------|---------|
| 健康记录查看/录入/编辑/删除 | 完全离线 |
| 家庭成员管理 | 完全离线 |
| 本地通知（用药/随访提醒） | 完全离线 |
| HealthKit 数据读取 | 本地 API |
| OCR 识别（Apple Vision 框架） | 完全离线（本地处理） |
| AI 解读（Claude API） | 需要网络 |
| iCloud 同步 | 需要网络（v1.1 规划） |

### 7.3 iOS 版本兼容性

- **最低支持版本：** iOS 17.0+（iPhone 12 及以上）
- **理由：** SwiftData 要求 iOS 17；Swift 6 严格并发模型；HealthKit 新 API；Swift Charts 成熟度

### 7.4 性能要求

- 应用冷启动（不含认证）< 2 秒
- 健康记录列表加载 < 0.5 秒（支持最多 10,000 条记录）
- Claude API 限流：令牌桶算法，每分钟最多 60 次请求；超时 30 秒后提示重试
- 图片懒加载（ImageStorageService），列表滑动帧率保持 60fps
