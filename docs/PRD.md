# AI Health Vault iOS App — 产品需求文档（PRD）

**版本：** v1.0
**日期：** 2026-03-25
**作者：** Product Manager
**状态：** 草稿，待团队评审
**关联 Issues：** AIH-9（本文档）、AIH-7（改造计划）、AIH-6（参考项目分析）

---

## 1. 产品愿景

**一句话：** 让家庭健康数据管理像记笔记一样简单，让 AI 成为你的私人健康顾问——数据始终留在你手中。

**核心价值主张：**
- 数据主权：所有健康数据默认存储在设备本地，用户掌控自己的数据
- 家庭健康：一个设备，管理全家健康档案
- AI 赋能：将医学术语变成普通人能看懂、能行动的建议

---

## 2. 目标用户

| 用户类型 | 特征 | 核心痛点 |
|---------|------|---------|
| 家庭健康管理者 | 30-50岁，负责管理家庭成员健康 | 体检报告看不懂，历史记录散乱 |
| 慢病患者/家属 | 需要长期追踪多项健康指标 | 指标变化趋势难以掌握，就诊前准备繁琐 |
| 健康意识较强的年轻人 | 关注日常健康数据，有 Apple Watch | HealthKit 数据零散，无综合分析 |

---

## 3. 关键决策说明

在制定本 PRD 前，需明确以下三个关键决策：

### 3.1 家庭成员管理是否进入 MVP？

**决策：✅ 进入 MVP（Phase 1）**

理由：
- 家庭管理是本产品的核心差异化定位，缺失则产品价值大幅削弱
- 技术实现代价低（SwiftData 多实体关系）
- 若 Phase 1 不包含，Phase 2 添加会需要大量数据迁移工作
- 接受标准：支持最少 1 个、最多 10 个家庭成员档案的创建、编辑和删除

### 3.2 Face ID 认证 vs 简单 PIN 码？

**决策：✅ Face ID / Touch ID + 备选 6 位 PIN**

理由：
- iOS 的 LocalAuthentication 框架原生支持生物认证，开发成本极低
- 健康数据敏感性高，生物认证是行业最佳实践（参考医疗 App）
- PIN 作为生物认证失败时的降级方案，保证可用性
- 不支持简单密码（太弱），不支持强制密码（摩擦太大）

### 3.3 AI 功能是否在 Phase 1 就有基础版本？

**决策：❌ Phase 1 不包含 AI 功能**

理由：
- Phase 1 目标是完整的本地数据管理闭环，AI 是增强层而非基础层
- 过早引入 Claude API 会增加 Phase 1 的风险和复杂度
- 用户必须先有数据，AI 才有分析价值
- AI 功能统一在 Phase 3 引入，避免功能碎片化
- **例外：** 相机 OCR pipeline（Phase 2）是 AI 的前置能力，按计划推进

---

## 4. MVP 范围确认（Phase 1 最小可用功能集）

Phase 1 的 MVP 是"能用的本地健康档案管理 App"，核心衡量标准：
> **用户能否在没有网络的情况下完整管理一个家庭成员的健康档案？**

### Phase 1 必须有（P0）

1. 应用初始化 + 本地生物认证（Face ID / Touch ID / PIN）
2. 家庭成员管理（创建、编辑、删除、切换）
3. 健康档案基础信息（基本信息 Profile）
4. 健康记录录入（手动输入，支持至少 4 种记录类型：体检记录、用药记录、就医记录、日常追踪）
5. 健康记录查看列表（按时间排序，支持按记录类型筛选）
6. 数据本地持久化（SwiftData，应用重启数据不丢失）

### Phase 1 明确不包含

- AI 功能（Phase 3）
- 相机 OCR（Phase 2）
- HealthKit 集成（Phase 2）
- iCloud 同步（Phase 4）
- 推送通知（Phase 4）
- 数据导出（Phase 4）

## 5. 用户故事清单

### Phase 1：Foundation（第 1-2 周）

#### US-1.1 应用认证
**As a** 用户，
**I want** 首次打开 App 时设置认证方式，之后每次打开需通过认证，
**So that** 我的家庭健康数据不会被他人访问。

**验收标准：**
- [ ] 首次启动引导用户开启 Face ID / Touch ID 认证
- [ ] 若设备不支持生物认证，提供 6 位 PIN 设置
- [ ] 生物认证失败 3 次后自动降级为 PIN 输入
- [ ] 认证成功后进入主界面，失败则留在锁屏
- [ ] App 进入后台超过 5 分钟后重新进入需再次认证
- [ ] 认证界面不显示任何健康数据

**优先级：** P0

---

#### US-1.2 家庭成员管理
**As a** 家庭健康管理者，
**I want** 为每个家庭成员创建独立的健康档案，
**So that** 我可以在一个 App 中管理全家人的健康数据，不会混淆。

**验收标准：**
- [ ] 可创建新的家庭成员档案（必填：姓名；可选：生日、血型、性别、关系）
- [ ] 支持为成员设置头像（选择颜色/图标）
- [ ] 可编辑成员信息
- [ ] 可删除成员（需二次确认，关联数据一并删除）
- [ ] 主界面顶部展示成员选择器，点击可切换当前成员
- [ ] 支持最多 10 个家庭成员
- [ ] 成员切换后所有数据视图更新为对应成员的数据

**优先级：** P0

---

#### US-1.3 健康档案基本信息
**As a** 用户，
**I want** 为家庭成员维护基本健康信息，
**So that** 我有一个完整的健康背景信息可以查阅，也方便就医时快速提供。

**验收标准：**
- [ ] 支持录入：ABO 血型、Rh 血型、身高、体重（手动录入）
- [ ] 支持录入既往病史（文本列表）
- [ ] 支持录入过敏史（文本列表）
- [ ] 支持录入慢性病/长期服药记录（文本列表）
- [ ] 信息随时可编辑
- [ ] "基本信息"作为档案的固定 Tab 展示

**优先级：** P0

---

#### US-1.4 体检记录录入与查看
**As a** 用户，
**I want** 手动录入体检报告的关键指标，
**So that** 我能在 App 中留存历史体检数据，方便对比。

**验收标准：**
- [ ] 可创建一条体检记录（包含：日期、机构名称、备注）
- [ ] 可在体检记录中添加多个指标条目（指标名称、数值、单位、参考范围）
- [ ] 可标记指标状态：正常/偏高/偏低/异常
- [ ] 体检记录列表按日期倒序排列
- [ ] 点击体检记录可查看详情（所有指标条目）
- [ ] 可编辑和删除体检记录

**优先级：** P0

---

#### US-1.5 用药记录管理
**As a** 用户，
**I want** 记录家庭成员正在服用或曾经服用的药物，
**So that** 就医时能快速提供用药历史，也避免药物重复或冲突。

**验收标准：**
- [ ] 可创建用药记录（药品名、剂量、频次、开始时间、结束时间、开药医生、适应症）
- [ ] 支持标记药物状态：进行中 / 已停用
- [ ] 用药记录列表区分"当前用药"和"历史用药"
- [ ] 可编辑和删除用药记录

**优先级：** P0

---

#### US-1.6 就医记录
**As a** 用户，
**I want** 记录每次就医的情况，
**So that** 我有完整的就医历史，方便复诊时向医生提供背景信息。

**验收标准：**
- [ ] 可创建就医记录（日期、医院、科室、医生、主诉、诊断、处置建议、下次随访日期）
- [ ] 可添加文字备注
- [ ] 就医记录列表按日期倒序排列
- [ ] 可编辑和删除就医记录

**优先级：** P0

---

#### US-1.7 日常追踪
**As a** 用户，
**I want** 手动记录日常健康数据（如血压、血糖、体重），
**So that** 我能追踪慢性指标的变化趋势。

**验收标准：**
- [ ] 支持追踪类型：血压（收缩压/舒张压）、血糖（空腹/餐后）、体重、心率、体温
- [ ] 每条记录包含：数值、时间、备注
- [ ] 同类型数据按时间排列（最新在前）
- [ ] 可编辑和删除追踪记录

**优先级：** P0

---

#### US-1.8 主导航结构
**As a** 用户，
**I want** 有清晰的导航结构访问不同功能，
**So that** 我能快速找到我需要的功能。

**验收标准：**
- [ ] 底部 TabView：健康档案 / 记录 / （预留：AI 分析）/ 设置
- [ ] 健康档案 Tab：展示当前成员基本信息
- [ ] 记录 Tab：展示所有记录类型的列表，可按类型筛选
- [ ] 设置 Tab：家庭成员管理、认证设置、App 信息
- [ ] 所有界面适配 iPhone 竖屏，支持 Dynamic Type

**优先级：** P0

---

### Phase 2：Core Health Records（第 3-4 周）

#### US-2.1 相机拍照上传体检报告
**As a** 用户，
**I want** 拍照或从相册选择体检报告图片并附加到记录中，
**So that** 我保留原始报告存档，也为后续 AI 解读做准备。

**验收标准：**
- [ ] 可在体检记录中附加图片（相机拍摄 / 相册选取）
- [ ] 支持多张图片
- [ ] 图片存储在本地，不上传云端
- [ ] 图片可全屏查看、缩放
- [ ] 可删除已附加图片

**优先级：** P0

---

#### US-2.2 OCR 识别体检数据（AI Pipeline 前置）
**As a** 用户，
**I want** 拍照后 App 自动识别报告中的关键数值，
**So that** 我不需要手动逐条录入，节省大量时间。

**验收标准：**
- [ ] 拍照后触发 Vision/OCR 识别
- [ ] 识别结果展示为可编辑列表（用户确认后才保存）
- [ ] 用户可修正识别错误的字段
- [ ] 无法识别的字段留空，提示用户手动填写
- [ ] 识别过程有加载态提示

**优先级：** P1（Phase 2 内交付）

---

#### US-2.3 HealthKit 数据读取
**As a** Apple Watch 用户，
**I want** 授权后 App 自动同步我的 HealthKit 数据，
**So that** 我不需要手动录入步数、心率、睡眠等数据。

**验收标准：**
- [ ] 首次进入追踪功能时请求 HealthKit 权限（步数、心率、睡眠、活动）
- [ ] 获得授权后自动拉取最近 30 天历史数据
- [ ] HealthKit 数据在日常追踪列表中展示，标注数据来源
- [ ] 后续新数据按后台定期刷新（非实时）
- [ ] 若用户拒绝授权，只显示手动录入数据

**优先级：** P0（Phase 2 内交付）

---

#### US-2.4 数据搜索
**As a** 用户，
**I want** 在所有健康记录中快速搜索关键词，
**So that** 我能找到某次特定就医记录或某个指标的历史数据。

**验收标准：**
- [ ] 全局搜索框（记录 Tab 顶部）
- [ ] 搜索范围：所有记录类型的标题、备注、指标名称
- [ ] 搜索结果高亮显示匹配关键词
- [ ] 实时搜索（输入即搜索，无需点击确认）

**优先级：** P1

---

### Phase 3：AI Intelligence（第 5-6 周）

#### US-3.1 体检报告 AI 解读
**As a** 用户，
**I want** 让 AI 解读我的体检报告指标，用通俗语言告诉我哪些需要关注，
**So that** 我不需要自己查资料，能快速理解健康状况。

**验收标准：**
- [ ] 体检记录详情页有"AI 解读"按钮
- [ ] 触发后调用 Claude API，传入指标数据（不包含图片）
- [ ] AI 返回：整体评估、需要关注的指标及原因、建议
- [ ] 解读结果可保存到记录中
- [ ] 调用 API 时明确告知用户"数据将用于 AI 分析"
- [ ] 网络不可用时提示用户，功能禁用

**优先级：** P0（Phase 3 内交付）

---

#### US-3.2 健康趋势分析
**As a** 用户，
**I want** 查看某个健康指标随时间的变化趋势图，
**So that** 我能直观了解健康状况是改善还是恶化。

**验收标准：**
- [ ] 日常追踪数据支持折线图展示（Swift Charts）
- [ ] 可选择时间范围（7天 / 30天 / 90天 / 自定义）
- [ ] 图表展示正常参考范围区域
- [ ] 体检记录同类指标跨时间对比图
- [ ] 异常点用颜色标注

**优先级：** P0

---

#### US-3.3 就诊准备助手
**As a** 用户，
**I want** 就医前让 AI 帮我整理症状和历史数据，生成就诊摘要，
**So that** 我在有限的就诊时间内能高效地告诉医生我的情况。

**验收标准：**
- [ ] 用户输入本次就诊原因/症状
- [ ] AI 结合历史记录生成就诊摘要（主诉、相关病史、当前用药、近期指标变化）
- [ ] 结果可复制为文本或分享
- [ ] 支持用户在发送给 AI 前预览哪些数据会被使用

**优先级：** P1

---

#### US-3.4 健康数据通俗解读
**As a** 用户，
**I want** 把复杂的医学指标解读发给家里的老人看，
**So that** 他们也能理解自己的健康状况。

**验收标准：**
- [ ] AI 解读结果支持"通俗版"输出（减少专业术语）
- [ ] 通俗解读可通过系统分享功能发送（微信、短信等）
- [ ] 分享内容不包含原始数值（可选是否包含）

**优先级：** P1

- [ ] 提醒通过本地推送通知

#### US-4.3 iCloud 家庭同步（可选）
**As a** 用户，
**I want** 可选地开启 iCloud 同步，

- [ ] 支持导出某成员的健康摘要 PDF（基本信息、近期体检记录、当前用药）

- [ ] 通过系统分享功能分享（打印/邮件/存储）

## 7. 技术约束说明

### 7.1 隐私与数据安全

| 约束 | 要求 |
|------|------|
| 数据存储 | 所有健康数据默认存储在设备本地（SwiftData），不上传任何服务器 |
| AI 分析数据传输 | 调用 Claude API 时，仅传送必要的结构化数据（指标名称+数值），不传原始图片，不传用户身份信息 |
| 用户告知 | 首次使用 AI 功能时，展示数据使用说明，用户需主动确认 |
| iCloud 同步 | 默认关闭，用户主动开启，且需加密传输 |
| 认证保护 | 应用层使用 LocalAuthentication 框架，数据层使用 iOS 文件系统加密（Data Protection） |

### 7.2 离线能力

| 功能 | 离线可用 |
|------|---------|
| 健康记录查看/录入/编辑/删除 | 完全离线 |
| 家庭成员管理 | 完全离线 |
| 本地通知（用药/随访提醒） | 完全离线 |
| HealthKit 数据读取 | 本地 API |
| AI 解读（Claude API） | 需要网络 |
| iCloud 同步 | 需要网络 |
| OCR 识别 | 使用 Vision 框架，本地处理 |

### 7.3 iOS 版本兼容性

- **最低支持版本：** iOS 17.0+
- **理由：** SwiftData 在 iOS 17 引入；Swift Charts 在 iOS 16 引入但在 17 更成熟；HealthKit 新 API 要求 17+
- **不支持：** iPhone 11 以下（iOS 17 要求至少 iPhone 12 内核）

### 7.4 性能要求

- 应用冷启动（不含认证）< 2 秒
- 健康记录列表加载 < 0.5 秒（支持最多 10,000 条记录）
- AI 解读等待时有加载态，超时（30 秒）后提示重试
- 图片附件使用懒加载，列表滑动帧率保持 60fps

## 8. 非功能性需求

### 8.1 可访问性
- 支持 VoiceOver（重要操作需提供 accessibility label）
- 支持 Dynamic Type（文字大小随系统设置调整）
- 颜色标注不作为唯一信息表达方式（需配合图标/文字）

### 8.2 本地化
- Phase 1 仅支持中文简体
- 代码使用 SwiftUI 的 LocalizedStringKey，为后续多语言预留结构
